-- ================================================================
-- VIEW SLICE
-- DAW-specific interaction/projection semantics.
-- This slice references Editor semantics and lowers to TerraUI Decl.
-- It is not the generic UI compiler; TerraUI owns Decl -> Bound -> Plan
-- -> Kernel. View only specifies what DAW views exist, what they point at,
-- which anchors they expose, and which Editor commands they may emit.
-- ================================================================
module View {

    -- Design axioms:
    --   • View references Editor objects; it does not duplicate them.
    --   • View owns application-specific interaction semantics, not generic
    --     UI layout/rendering semantics.
    --   • View -> TerraUI lowering should be total in application/runtime use:
    --     return a valid degraded TerraUI subtree plus diagnostics rather than
    --     tearing down the shell when one surface is incomplete.
    --   • Semantic ids come from Editor. TerraUI keys are derived from them
    --     during lowering to TerraUI Decl.
    --   • Anchors are local visual targets, not semantic object identity.
    --   • View-local state (tabs, scroll memory, split ratios, collapse) may
    --     live here; musical/project truth must stay in Editor.
    --   • View is normally derived from Editor + session state; it is not the
    --     canonical saved project document in the way Editor is.
    --   • TerraUI action dispatch must come back to either Editor command
    --     semantics or explicit View/session-state updates; it must never
    --     bypass them with ad hoc widget-local mutation.
    --   • A thing merits its own View type when it is a stable user-visible
    --     surface, subtree, or repeated projection with its own identity,
    --     anchors, commands, or lowering contract.
    --   • Do not model transient rendering details here: hover, pixel rects,
    --     drag ghosts, scroll physics, and raw draw commands stay out.
    --
    -- Ref vs id note:
    --   • when View points at a real stable Editor-backed object, prefer a
    --     typed ref over a naked id
    --   • refs are ontology-level handles: they encode object kind and, when
    --     needed, ownership path/scope
    --   • ids/indexes remain appropriate for structural/local payload such as
    --     port ids, point indexes, route indexes, slot positions, and insert
    --     positions
    --   • do not invent fake refs for purely local structure; use refs only
    --     when there is a real semantic object being addressed
    --
    -- View -> TerraUI lowering note:
    --   • View types define semantic UI surfaces/subtrees; TerraUI Decl defines
    --     the concrete render/layout tree
    --   • one View node usually lowers to one TerraUI subtree, not necessarily
    --     to one single TerraUI node
    --   • lowering methods should be total: if a surface-specific lowering is
    --     incomplete, emit a stable keyed error/placeholder subtree and record
    --     diagnostics instead of failing the whole window
    --   • View.Identity should lower into stable TerraUI ids/keys
    --   • typed anchors should lower into stable local TerraUI node ids/parts
    --     used for hit targets, floating attachment, and internal targeting
    --   • typed commands should lower so `action_id` becomes TerraUI
    --     Input.action, while the typed command payload remains the DAW-side
    --     dispatch contract
    --   • repeated *View arrays normally lower to repeated keyed child
    --     subtrees; sum types normally lower by variant/pattern match
    --   • View should specify semantic structure and interaction contracts,
    --     not pixel layout mechanics or renderer-specific widget trivia

    -- Root is the application view tree for the current editor session.
    -- Construction policy:
    --   • usually derived from Editor.Project + current app/session context
    --   • may be cached or partially persisted as workspace/session state
    --   • should not become the authoritative project save format
    Root = (
        View.Shell         shell,
        View.Focus         focus,
        View.SessionState* session_state
    ) unique
    methods {
        to_decl(ViewCtx ctx) -> TerraUIDecl
    }

    -- SemanticRef names stable Editor-backed objects as they appear in View.
    -- Scoped constructors intentionally encode ownership where object ids are
    -- not globally meaningful on their own: container_id is the owning
    -- container device id, note refs are clip-scoped, slot/send refs are
    -- track-scoped, and ParamRef uses the full Editor.ParamOwnerRef path
    -- rather than a bare owner number.
    SemanticRef
        = ProjectRef
        | TrackRef(number track_id)
        | DeviceRef(number device_id)
        | LayerRef(number container_id, number layer_id)
        | SelectorBranchRef(number container_id, number branch_id)
        | SplitBandRef(number container_id, number band_id)
        | GridModuleRef(number device_id, number module_id)
        | ClipRef(number clip_id)
        | NoteRef(number clip_id, number note_id)
        | SceneRef(number scene_id)
        | SlotRef(number track_id, number slot_index)
        | SendRef(number track_id, number send_id)
        | ParamRef(Editor.ParamOwnerRef owner_ref, number param_id)
        | ModulatorRef(number device_id, number modulator_id)

    -- ChainRef canonical encoding guidance for TerraUI key derivation:
    --   TrackChain(7)                -> "track_chain/7"
    --   DeviceNoteFX(42)             -> "device_note_fx/42"
    --   DevicePostFX(42)             -> "device_post_fx/42"
    --   LayerChain(100,3)            -> "layer_chain/100/3"
    --   SelectorBranchChain(120,2)   -> "selector_chain/120/2"
    --   SplitBandChain(130,1)        -> "split_chain/130/1"
    ChainRef
        = TrackChain(number track_id)
        | DeviceNoteFX(number device_id)
        | DevicePostFX(number device_id)
        | LayerChain(number container_id, number layer_id)
        | SelectorBranchChain(number container_id, number branch_id)
        | SplitBandChain(number container_id, number band_id)

    Shell = (
        View.TransportBar transport,
        View.MainArea     main_area,
        View.Sidebar*     sidebars,
        View.StatusBar?   status_bar
    )
    methods {
        to_decl(ViewCtx ctx) -> TerraUIDecl
    }

    -- MainArea is the high-level workspace composition algebra.
    -- It stays as a sum rather than a richer product type because the primary
    -- design choice here is which major DAW work surfaces are present together,
    -- not a uniform set of child slots shared by every workspace mode.
    MainArea
        = ArrangementMain(View.ArrangementView arrangement,
                          View.DetailPanel? detail_panel)
        | LauncherMain(View.LauncherView launcher,
                       View.DetailPanel? detail_panel)
        | MixerMain(View.MixerView mixer,
                    View.DetailPanel? detail_panel)
        | HybridMain(View.ArrangementView arrangement,
                     View.LauncherView launcher,
                     View.MixerView mixer,
                     View.DetailPanel? detail_panel)

    -- Sidebar remains a sum because browser and inspector are semantically
    -- distinct surfaces that happen to share shell placement.
    Sidebar
        = BrowserSidebar(View.BrowserView browser)
        | InspectorSidebar(View.InspectorView inspector)

    -- StatusBar remains simple because it is primarily a readout surface.
    -- It merits its own type because it is a stable global shell region with a
    -- distinct lowering contract, but it does not yet need a richer command
    -- family.
    StatusBar = (
        string              left_text,
        string?             center_text,
        string?             right_text,
        View.Identity       identity,
        View.StatusAnchor*  anchors
    )

    StatusAnchor = (
        View.StatusAnchorKind kind
    )

    StatusAnchorKind
        = StatusRootA
        | StatusLeftA
        | StatusCenterA
        | StatusRightA

    -- Focus separates semantic selection from workspace-local active surface.
    -- This keeps "what object is selected" distinct from "which editor is
    -- currently receiving interaction".
    Focus = (
        View.Selection     selection,
        View.ActiveSurface active_surface
    )

    Selection
        = NoSelection
        | SelectedTrack(View.SemanticRef track_ref)
        | SelectedDevice(View.SemanticRef device_ref)
        | SelectedClip(View.SemanticRef clip_ref)
        | SelectedNote(View.SemanticRef note_ref)
        | SelectedScene(View.SemanticRef scene_ref)
        | SelectedSlot(View.SemanticRef slot_ref)
        | SelectedGridModule(View.SemanticRef module_ref)
        | SelectedModulator(View.SemanticRef modulator_ref)

    ActiveSurface
        = ArrangementSurface
        | LauncherSurface
        | MixerSurface
        | BrowserSurface
        | DeviceSurface(View.SemanticRef device_ref)
        | GridSurface(View.SemanticRef device_ref)
        | PianoRollSurface(View.SemanticRef clip_ref)
        | InspectorSurface

    -- SessionState is explicitly UI-local/workspace-local state.
    -- It may be persisted separately from Editor.Project if desired.
    SessionState
        = SplitRatioState(string key, number value)
        | ScrollState(string key, number x, number y)
        | TabState(string key, string active_tab)
        | CollapseState(string key, boolean open)

    -- Review note: SessionState carries workspace memory only. If a proposed
    -- field is meaningful to DSP, arrangement semantics, launcher behavior, or
    -- project save/load, it belongs in Editor/Authored rather than here.

    -- TrackHeaderView is the shared track-identity/control projection reused
    -- across arrangement, launcher, and mixer contexts. It exists because the
    -- user recognizes "the track header" as a stable DAW object even when the
    -- surrounding work surface changes.
    TrackHeaderView = (
        View.SemanticRef          track_ref,
        View.TrackHeaderRole      role,
        View.Identity             identity,
        View.TrackHeaderAnchor*   anchors,
        View.TrackHeaderCommand*  commands
    )

    TrackHeaderRole
        = ArrangementHeaderRole
        | LauncherHeaderRole
        | MixerHeaderRole

    TrackHeaderAnchor = (
        View.TrackHeaderAnchorKind kind
    )

    TrackHeaderAnchorKind
        = TrackHeaderRootA
        | TrackHeaderTitleA
        | TrackHeaderColorA
        | TrackHeaderMuteA
        | TrackHeaderSoloA
        | TrackHeaderArmA
        | TrackHeaderMonitorA
        | TrackHeaderMeterA
        | TrackHeaderFoldA
        | TrackHeaderIOA

    TrackFlagKind
        = TrackMutedF
        | TrackSoloedF
        | TrackArmedF
        | TrackMonitorInputF
        | TrackPhaseInvertF

    TrackHeaderCommand = (
        string action_id,
        View.TrackHeaderCommandKind kind,
        View.SemanticRef       track_ref,
        View.TrackFlagKind?    flag,
        boolean?               bool_value,
        View.SemanticRef?      target_track_ref
    )

    -- THCCSetTrackFlag / THCCSetTrackOutput target Editor.Track semantics.
    -- `track_ref` should normally be View.TrackRef(track_id); `target_track_ref`
    -- should normally be View.TrackRef(target_track_id) when present.
    -- THCCToggleFold / THCCSelectTrack are view/workspace actions.
    TrackHeaderCommandKind
        = THCCSetTrackFlag
        | THCCToggleFold
        | THCCSetTrackOutput
        | THCCSelectTrack

    -- TransportBar is the typed global transport/status surface.
    -- It is view-level and application-facing: play state, tempo editing, loop,
    -- and quantize affordances live here as commandable UI targets rather than
    -- as ad hoc toolbar conventions.
    TransportBar = (
        boolean                show_tempo,
        boolean                show_time_sig,
        boolean                show_loop,
        boolean                show_quantize,
        View.Identity          identity,
        View.TransportAnchor*  anchors,
        View.TransportCommand* commands
    )
    methods {
        to_decl(ViewCtx ctx) -> TerraUIDecl
    }

    TransportAnchor = (
        View.TransportAnchorKind kind
    )

    TransportAnchorKind
        = TransportRootA
        | TransportNavA
        | TransportPlayA
        | TransportStopA
        | TransportRecordA
        | TransportLoopA
        | TransportTempoA
        | TransportTimeSigA
        | TransportPositionA
        | TransportQuantizeA
        | TransportStatusA

    TransportCommand = (
        string action_id,
        View.TransportCommandKind kind,
        number? number_value,
        Editor.Quantize? quantize_value,
        boolean? bool_value
    )

    TransportCommandKind
        = TCCPlay
        | TCCStop
        | TCCToggleRecord
        | TCCToggleLoop
        | TCCSetTempo
        | TCCSetTimeSigNum
        | TCCSetTimeSigDen
        | TCCSetLaunchQuantize

    -- ArrangementView is the typed timeline surface for clips and automation
    -- on tracks. It stays view-level: semantic clip truth remains in
    -- Editor.Clip and automation truth remains in Editor.ParamSource curves.
    -- Construction policy:
    --   • visible_track_refs defines vertical ordering/filtering of track lanes
    --   • ruler/grid/playhead/loop_region/selection describe shared timeline
    --     chrome and editing overlays
    --   • lanes project visible per-track clip and automation regions
    --   • arrangement-local zoom/scroll still belongs in SessionState, not here
    ArrangementView = (
        View.SemanticRef*         visible_track_refs,
        View.Identity             identity,
        View.ArrangementAnchor*   anchors,
        View.ArrangementCommand*  commands,
        View.ArrangementRuler?       ruler,
        View.ArrangementGridView?    grid,
        View.ArrangementPlayheadView? playhead,
        View.ArrangementLoopRegionView? loop_region,
        View.ArrangementSelectionView? selection,
        View.ArrangementLane*     lanes
    )
    methods {
        to_decl(ViewCtx ctx) -> TerraUIDecl
    }

    -- ArrangementAnchor lowering guidance:
    --   • ArrangementRootA   -> local anchor like "root"
    --   • ArrangementRulerA  -> local anchor like "ruler"
    --   • ArrangementGridA   -> local anchor like "grid"
    --   • ArrangementCanvasA -> local anchor like "canvas"
    --   • `track_ref` should normally be View.TrackRef(track_id)
    --   • `clip_ref` should normally be View.ClipRef(clip_id)
    --   • ArrangementAutomation* anchors should encode (param_ref,point)
    --     deterministically within the arrangement subtree
    ArrangementAnchor = (
        View.ArrangementAnchorKind kind,
        View.SemanticRef? track_ref,
        View.SemanticRef? clip_ref,
        View.SemanticRef? param_ref,
        number? point_index
    )

    ArrangementAnchorKind
        = ArrangementRootA
        | ArrangementRulerA
        | ArrangementGridA
        | ArrangementCanvasA
        | ArrangementPlayheadA
        | ArrangementLoopA
        | ArrangementLaneA
        | ArrangementLaneHeaderA
        | ArrangementLaneBodyA
        | ArrangementClipBodyA
        | ArrangementClipLabelA
        | ArrangementClipLeftTrimA
        | ArrangementClipRightTrimA
        | ArrangementAutomationLaneA
        | ArrangementAutomationPointA
        | ArrangementSelectionA

    -- ArrangementCommand lowering guidance:
    --   • action_id should be deterministic and arrangement-surface scoped
    --   • `track_ref` / `clip_ref` / `param_ref` are semantic payloads, not
    --     UI-local ids
    --   • beat values are view-projection values used to dispatch semantic clip,
    --     automation, and transport commands back to Editor/app logic
    ArrangementCommand = (
        string action_id,
        View.ArrangementCommandKind kind,
        View.SemanticRef? track_ref,
        View.SemanticRef? clip_ref,
        View.SemanticRef? param_ref,
        number? point_index,
        number? start_beats,
        number? end_beats,
        number? lane,
        number? number_value,
        boolean? bool_value
    )

    -- ACCSetPlayhead / ACCToggleLoop / ACCSetLoopRange target Editor.Transport.
    -- ACCSetTimeSelection / ACCClearTimeSelection are view-local workspace
    -- actions. Clip and automation commands target Editor.Clip and
    -- Editor.ParamValue automation semantics.
    ArrangementCommandKind
        = ACCSetPlayhead
        | ACCToggleLoop
        | ACCSetLoopRange
        | ACCSetTimeSelection
        | ACCClearTimeSelection
        | ACCAddClip
        | ACCMoveClip
        | ACCResizeClipStart
        | ACCResizeClipEnd
        | ACCSelectClip
        | ACCAddAutomationPoint
        | ACCMoveAutomationPoint
        | ACCRemoveAutomationPoint
        | ACCSelectAutomationPoint

    -- ArrangementRuler is the shared time axis above the clip lanes.
    -- The semantic ref in identity will typically be ProjectRef under an
    -- arrangement-specific key_space; beat values describe only the current
    -- visible window, not authoritative transport state.
    ArrangementRuler = (
        number                   visible_start_beats,
        number                   visible_end_beats,
        View.Identity            identity,
        View.ArrangementAnchor*  anchors,
        View.ArrangementCommand* commands
    )

    -- ArrangementGridView is the repeated timing scaffold behind the lanes.
    -- It merits its own type because the grid is a stable visible projection
    -- with its own anchors and selection/editing relationship.
    ArrangementGridView = (
        number                   visible_start_beats,
        number                   visible_end_beats,
        View.Identity            identity,
        View.ArrangementAnchor*  anchors
    )

    -- ArrangementPlayheadView projects the current song position into
    -- arrangement space. It is a view-only marker, not a second transport
    -- model.
    ArrangementPlayheadView = (
        number                  position_beats,
        View.Identity           identity,
        View.ArrangementAnchor* anchors
    )

    -- ArrangementLoopRegionView projects the current arrangement loop
    -- brace/range. Loop semantics still come from Editor transport/project
    -- state.
    ArrangementLoopRegionView = (
        number                   start_beats,
        number                   end_beats,
        boolean                  enabled,
        View.Identity            identity,
        View.ArrangementAnchor*  anchors,
        View.ArrangementCommand* commands
    )

    -- ArrangementLane is one visible timeline lane for a semantic track.
    -- It is split into header and body so the ontology reflects the real DAW
    -- structure rather than flattening track identity and clip canvas together.
    ArrangementLane = (
        View.SemanticRef                track_ref,
        View.Identity                   identity,
        View.ArrangementLaneHeaderView  header,
        View.ArrangementLaneBodyView    body
    )

    ArrangementLaneHeaderView = (
        View.SemanticRef           track_ref,
        View.Identity              identity,
        View.TrackHeaderView       track_header,
        View.ArrangementAnchor*    anchors,
        View.ArrangementCommand*   commands
    )

    ArrangementLaneBodyView = (
        View.SemanticRef                   track_ref,
        View.Identity                      identity,
        View.ArrangementAnchor*            anchors,
        View.ArrangementCommand*           commands,
        View.ArrangementClipView*          clips,
        View.ArrangementAutomationLaneView* automation_lanes
    )

    -- Arrangement automation projects ParamValue automation curves that are
    -- already present in Editor.ParamSource(AutomationRef(...)). `param_ref`
    -- should normally be a View.ParamRef(Editor.ParamOwnerRef(...),param_id).
    ArrangementAutomationLaneView = (
        View.SemanticRef                  track_ref,
        View.SemanticRef                  param_ref,
        View.Identity                     identity,
        View.ArrangementAnchor*           anchors,
        View.ArrangementCommand*          commands,
        View.ArrangementAutomationPointView* points
    )

    ArrangementAutomationPointView = (
        View.SemanticRef           track_ref,
        View.SemanticRef           param_ref,
        number                     point_index,
        number                     time_beats,
        number                     value,
        View.Identity              identity,
        View.ArrangementAnchor*    anchors,
        View.ArrangementCommand*   commands
    )

    -- ArrangementSelectionView is view-local workspace state projected into
    -- the arrangement editor. It is intentionally not a semantic project
    -- object, so it carries no semantic ref of its own.
    ArrangementSelectionView = (
        number                   start_beats,
        number                   end_beats,
        View.Identity            identity,
        View.ArrangementAnchor*  anchors,
        View.ArrangementCommand* commands
    )

    -- ArrangementClipView is one rendered clip item within an arrangement lane.
    -- It references a semantic clip by id and exposes local anchors for body,
    -- label, and trim affordances while keeping command payloads semantic.
    ArrangementClipView = (
        View.SemanticRef          track_ref,
        View.SemanticRef          clip_ref,
        View.Identity             identity,
        View.ArrangementAnchor*   anchors,
        View.ArrangementCommand*  commands
    )

    -- PianoRollView is the native note-region editor for one note clip.
    -- It edits Editor.NoteRegion / Editor.Note semantics directly rather than
    -- exposing raw note-on/note-off event streams as the primary UI ontology.
    PianoRollView = (
        View.SemanticRef               clip_ref,
        View.Identity                  identity,
        View.PianoRollAnchor*          anchors,
        View.PianoRollCommand*         commands,
        View.PianoKeyboardView         keyboard,
        View.PianoGridView             grid,
        View.PianoPlayheadView?        playhead,
        View.PianoLoopRegionView?      loop_region,
        View.PianoSelectionView?       selection,
        View.PianoNoteView*            notes,
        View.PianoVelocityLaneView?    velocity_lane,
        View.PianoExprLaneView*        expr_lanes
    )
    methods {
        to_decl(ViewCtx ctx) -> TerraUIDecl
    }

    PianoKeyboardView = (
        number                  lowest_pitch,
        number                  highest_pitch,
        View.Identity           identity,
        View.PianoRollAnchor*   anchors,
        View.PianoRollCommand*  commands,

    PianoGridView = (

        number                  lowest_pitch,

        View.PianoRollAnchor*   anchors
    )

    PianoPlayheadView = (

        View.Identity           identity,
        View.PianoRollAnchor*   anchors
    )

    PianoLoopRegionView = (

    PianoSelectionView = (

    -- LauncherView is the typed session/launcher surface.
    -- It projects visible tracks and scenes into columns, scene headers, stop
    -- controls, and per-track slot cells with explicit command payloads.

        View.SemanticRef*        visible_track_refs,
        View.SemanticRef*        visible_scene_refs,
        View.Identity            identity,
        View.LauncherAnchor*     anchors,

        to_decl(ViewCtx ctx) -> TerraUIDecl

    -- Launcher anchors should prefer semantic refs over ad hoc coordinate-like
    -- ids. Typical normal forms are View.TrackRef(track_id),
    -- View.SceneRef(scene_id), View.SlotRef(track_id,slot_index), and
    -- View.ClipRef(clip_id) when present.
    LauncherAnchor = (
        View.LauncherAnchorKind kind,
        View.SemanticRef? track_ref,

    -- Identity is the declarative bridge from View-side stable identity to
    -- TerraUI instance identity. Lowering policy:
    --   • `ref` may identify a real Editor object, a chain scope, or a stable
    --     non-semantic browser/workspace key
    --   • key_space distinguishes multiple UI projections of the same target
    --   • TerraUI `key`/`scope` is derived from (key_space, ref)
    --   • derivation must be deterministic and path-stable for the same view
    --     projection so TerraUI local state persists correctly
    -- Canonical naming convention:
    --   • derive a TerraUI scope/key path as:
    --       <key_space>/<ref encoding>
    --   • ref encodings should be textual and unambiguous, e.g.:
    --       TrackRef(7)                    -> "track/7"
    --       DeviceRef(42)                  -> "device/42"
    --       LayerRef(100,3)                -> "layer/100/3"
    --       SelectorBranchRef(120,2)       -> "selector_branch/120/2"
    --       SplitBandRef(130,1)            -> "split_band/130/1"
    --       GridModuleRef(200,8)           -> "grid_module/200/8"
    --       ClipRef(55)                     -> "clip/55"
    --       NoteRef(55,12)                  -> "note/55/12"
    --       SceneRef(3)                     -> "scene/3"
    --       SlotRef(7,4)                    -> "slot/7/4"
    --       SendRef(7,2)                    -> "send/7/2"
    --       ParamRef(Editor.DeviceOwner(42),9) -> "param/device/42/9"
    --       TrackChain(7)                   -> "track_chain/7"
    --   • example full keys:
    --       "mixer_strip/track/7"
    --       "launcher_slot/slot/7/4"
    --       "device_chain/track_chain/7"
    --       "device_entry/device/42"
    --       "piano_note/note/55/12"
    --       "browser_item/preset://factory/super-bass"
    -- Typical examples:
    --   • ("mixer_strip", IdentitySemantic(TrackRef(7)))
    --       -> keyed strip instance for track 7
    --   • ("device_chain", IdentityChain(TrackChain(7)))
    --       -> keyed chain root for that scope
    --   • ("device_entry", IdentitySemantic(DeviceRef(42)))
    --       -> keyed card instance for device 42
    --   • ("browser_item", IdentityKey("preset://factory/super-bass"))
    --       -> keyed external browser row without inventing a fake semantic ref
    IdentityRef
        = IdentitySemantic(View.SemanticRef semantic_ref)
        | IdentityChain(View.ChainRef chain_ref)
        | IdentityKey(string stable_key)

    Identity = (
        string key_space,
        View.IdentityRef ref
    )

    -- Anchors are local named visual targets exposed intentionally by a view.
    -- They lower to TerraUI local anchors, not to Editor semantic ids.
    -- First-class surfaces should prefer typed anchor families; this generic
    -- record exists only as a fallback for broad or still-unspecialized views.
    -- At the current maturity of this module, most major DAW surfaces should
    -- not need to use it directly.
    -- Anchor names should be deterministic within the keyed subtree that owns
    -- them, so floating/tooltip/gesture targeting is stable.
    -- Canonical naming convention:
    --   • anchors are local leaf names inside a keyed subtree
    --   • simple singleton anchors use conventional names like:
    --       "root", "header", "title", "meter", "body", "canvas"
    --   • indexed anchors append semantic payload deterministically, e.g.:
    --       "insert_3", "drop_3", "send_12", "in_0", "out_1"
    --   • command payloads must never rely on anchor parsing alone; anchors
    --     are for local targeting, while semantic payload remains explicit
    Anchor = (
        string name,
        string purpose
