# Terra DAW Design Tokens v0.1

**Status:** Draft  
**Depends on:** `daw-design-system.md`  
**Purpose:** Define implementation-ready design tokens for the Terra DAW visual system

---

## 1. Overview

This document formalizes the token layer of the Terra DAW design system.

It translates the visual principles from `daw-design-system.md` into concrete token groups that can be mapped into:

- TerraUI theme values
- design tooling and visual mockups
- CSS variables, if needed in web-facing tooling
- future theme packs

These tokens do not assume that the runtime UI is rendered with SVG. SVG terminology may still appear in visual specs as a clean vector reference model.

The token system is organized in four layers:

1. base tokens
2. semantic tokens
3. component tokens
4. derived contextual tokens

---

## 2. Token Architecture

### 2.1 Base tokens

Base tokens define raw values.

Examples:

- spacing values
- radii
- stroke widths
- font sizes
- raw palette entries

These should rarely be used directly in components.

### 2.2 Semantic tokens

Semantic tokens map base tokens to meaning.

Examples:

- `surface.panel`
- `text.primary`
- `border.subtle`
- `focus.ring`

These should be the default tokens used by primitives.

### 2.3 Component tokens

Component tokens specialize semantic tokens for a specific control.

Examples:

- `button.neutral.bg.default`
- `knob.arc.value`
- `meter.fill.warning`

These should be used when a component needs stable explicit roles.

### 2.4 Derived contextual tokens

Derived tokens are computed from context.

Examples:

- track color variants
- selected clip border from track color
- muted track dimming rules
- modulation overlay alpha variants

These should be generated by theme logic rather than manually duplicated.

---

## 3. Naming Rules

### 3.1 General format

Use dot-separated, semantic-first naming.

Pattern:

`category.subcategory.role.state`

Examples:

- `surface.control.hover`
- `button.neutral.bg.default`
- `button.transport.play.active`
- `meter.fill.clip`
- `track.blue.fillSoft`

### 3.2 Naming principles

- prefer semantic meaning over visual description
- avoid tokens named after specific screens
- avoid ambiguous names like `primary2` or `gray7` in semantic layers
- keep state suffixes consistent across components

## 8. Track Color Derivation

Track colors should be authored as base hues, then derived into stable UI roles.

## 8.1 Required derived roles

For each track color `T`, generate:

- `track.<name>.base`
- `track.<name>.fillSoft`
- `track.<name>.fillStrong`
- `track.<name>.border`
- `track.<name>.textOn`
- `track.<name>.dimmed`
- `track.<name>.selectedRing`

## 8.2 Derivation rules

These are visual rules, not strict formulas yet:

- `fillSoft`: low-saturation, darker, panel-safe tint
- `fillStrong`: medium-saturation clip-safe fill
- `border`: brighter, cleaner stroke tone
- `textOn`: high-contrast text color selected for readability
- `dimmed`: desaturated lowered-contrast variant
- `selectedRing`: brighter, more luminous accent version

## 10. Vector Mapping Guidance

These rules keep tokens usable in any clean vector-based visual specification, including mockups and reference diagrams.

### 10.1 Fills

Use semantic or component tokens for fills rather than raw colors.

### 10.2 Strokes

Use explicit border tokens rather than ad hoc line colors.

### 10.3 Focus

Focus should usually be a dedicated ring element, not a modified main border.

### 10.4 Modulation

Modulation should always use a dedicated token family and never share default value color.

## 12. Rules for Future Token Expansion

When adding tokens:

- add a semantic token before adding a component-specific override
- avoid duplicating raw values unnecessarily
- preserve state naming consistency
- reserve new semantic colors only for genuinely distinct meaning
- do not create one-off tokens for a single screen unless absolutely required
